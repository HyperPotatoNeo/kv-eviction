=============================================================
THE CHAINED-ADMISSION RoPE COLLISION BUG — A SIMPLE EXAMPLE
=============================================================

Setup: 128-token sequence, block_size = 16 (so 8 blocks total).
       Each turn has 32 tokens (= 2 blocks).
       max_turns = 2 → keep most recent 2 turn pairs + sys + new user.
       sys = 16 tokens (block 0).

Tokens 1..128 represent a chat conversation laid out like this:

  Block 0  (tokens   1.. 16): system prompt
  Block 1  (tokens  17.. 32): turn 0 user
  Block 2  (tokens  33.. 48): turn 0 assistant
  Block 3  (tokens  49.. 64): turn 1 user
  Block 4  (tokens  65.. 80): turn 1 assistant
  Block 5  (tokens  81.. 96): turn 2 user
  Block 6  (tokens  97..112): turn 2 assistant
  Block 7  (tokens 113..128): turn 3 user (just submitted)

Each block holds 16 K vectors. RoPE rotates each K at its
"logical position" when written. Logical position is FROZEN
once written — you cannot re-rotate a stored K.


-----------------------------------------------------------
TURN 1: no admission, fresh writes (logical positions = physical)
-----------------------------------------------------------

Request submits 32 tokens (sys + T0_user). vLLM writes K at:

  Slot   0..15  (K of tokens   1.. 16): logical positions  0..15
  Slot  16..31  (K of tokens  17.. 32): logical positions 16..31

position_offset = 0. Cache is dense, monotonic.

Decode produces T0_assistant (tokens 33..48). New K appended:

  Slot  32..47  (K of tokens  33.. 48): logical positions 32..47


-----------------------------------------------------------
TURN 2: still no admission, prefix cache hit
-----------------------------------------------------------

User submits a new request with prompt:
  sys + T0 + T1_user = 64 tokens

Prefix cache hits the first 48 tokens (= turn 1's full state).
The 16 new tokens (T1_user) are prefilled.

After prefill + decode:

  Slot   0..15: K at logical  0..15  (sys, inherited)
  Slot  16..31: K at logical 16..31  (T0 user, inherited)
  Slot  32..47: K at logical 32..47  (T0 asst, inherited)
  Slot  48..63: K at logical 48..63  (T1 user, new prefill)
  Slot  64..79: K at logical 64..79  (T1 asst, decoded)

position_offset = 0. Cache still dense and monotonic.


-----------------------------------------------------------
TURN 3: FIRST ADMISSION (max_turns=2, evict T0)
-----------------------------------------------------------

User submits prompt:
  sys + T0 + T1 + T2_user = 96 tokens

vLLM scheduler decides: keep at most 2 turn pairs, so EVICT T0.
That means free blocks 1, 2 (= 32 tokens). Bump position_offset
by 32 so future writes don't collide with what's left in cache.

BEFORE admission (after prefix cache hit on 80 tokens):

  Slot   0..15: K at logical  0..15  (sys)
  Slot  16..31: K at logical 16..31  (T0 user)  ← will be evicted
  Slot  32..47: K at logical 32..47  (T0 asst)  ← will be evicted
  Slot  48..63: K at logical 48..63  (T1 user)
  Slot  64..79: K at logical 64..79  (T1 asst)

After admission splice (drop slots 16..47 = T0):

  Slot   0..15: K at logical  0..15  (sys)
  Slot  16..31: K at logical 48..63  (T1 user, shifted down)
  Slot  32..47: K at logical 64..79  (T1 asst, shifted down)

position_offset bumped: 0 → 32 (= tokens_evicted)

Now the scheduler tells the worker: prefill the new tokens
(T2_user at submitted positions 80..95) at physical positions
48..63 in this request's cache. Worker writes K at logical:

  physical + position_offset = 48 + 32 = 80  →  K at logical 80
  physical + position_offset = 63 + 32 = 95  →  K at logical 95

After prefill of T2_user:

  Slot   0..15: K at logical  0..15  (sys)
  Slot  16..31: K at logical 48..63  (T1 user)
  Slot  32..47: K at logical 64..79  (T1 asst)
  Slot  48..63: K at logical 80..95  (T2 user, new prefill)

Decode produces T2_asst (16 tokens) at logical 96..111.

Final turn 3 cache:

  Slot   0..15:  logical  0..15
  Slot  16..31:  logical 48..63
  Slot  32..47:  logical 64..79
  Slot  48..63:  logical 80..95
  Slot  64..79:  logical 96..111

Cache offsets to recover logical from physical:
  Slot range  | logical_start | offset (logical - physical)
   0..15      |    0          |  0    ← sys K rotated at logical 0..15
                                       (offset that WOULD recover its
                                       logical positions if Q used it)
  16..79      |   48..96      | 32    ← all post-sys K rotated at logical 48..111

So WITHIN one request, the "natural" offset varies:
  - sys K rotations imply offset=0
  - post-sys K rotations imply offset=32

THE IMPORTANT NUANCE: vLLM currently uses ONE `position_offset`
per request and applies it UNIFORMLY to every token's Q. So at
turn 3, Q at physical 0..15 (sys positions) ends up rotating at
logical 32..47 — not the natural 0..15 of sys K. The model sees
sys K at relative positions 32..17 from Q instead of 0..-15.

That's a SHIFT, not a collision. The model copes because the
relative offsets are still within its trained range. The bug is
NOT here. The actual bug is what happens when a SECOND admission
fires on top of inherited K from this turn 3 — that's where two
K end up at THE SAME logical position. See turn 4 below.

(So to your question: yes, sys ideally shouldn't be offset.
Current vLLM does offset it uniformly, causing a tolerable shift.
The 2-piece position_offset refactor in
plans/piecewise_position_offset.md would fix this so sys keeps
offset=0.)


-----------------------------------------------------------
TURN 4: SECOND ADMISSION — THIS IS WHERE IT BREAKS
-----------------------------------------------------------

User submits prompt:
  sys + T1 + T2 + T3_user = 96 tokens

(Note: T0 was already evicted at turn 3. T1, T2 are still kept.
 T3_user is the new user message we just submitted.)

This is a FRESH request from vLLM's perspective (each chat()
call is a new request). position_offset starts at 0.

Prefix cache hit: this request's submitted prompt prefix
matches what turn 3 stored, so vLLM reuses turn 3's blocks.

Let's say cache hit covers 64 tokens (= 4 blocks). The
inherited blocks ARE TURN 3'S blocks — same KVCacheBlock
objects, same K vectors with the same FROZEN RoPE rotations:

  Slot   0..15: K at logical  0..15  (sys, original from turn 1)
  Slot  16..31: K at logical 48..63  (T1 user, frozen at logical 48..63)
  Slot  32..47: K at logical 64..79  (T1 asst, frozen at logical 64..79)
  Slot  48..63: K at logical 80..95  (T2 user, frozen at logical 80..95)

The inheriting request (turn 4) didn't write these K — it
inherits them via prefix cache hit. Their logical positions
ARE WHAT THEY ARE (frozen). Turn 4 cannot move them.

vLLM scheduler decides: evict T1 (the oldest kept turn). That's
slots 16..47 of turn 4's cache (= 32 tokens). Bump position_offset
by 32.

After admission splice:

  Slot   0..15: K at logical  0..15  (sys, kept)
  Slot  16..31: K at logical 80..95  (was T2 user at slot 48..63,
                                       shifted down by 32)

position_offset bumped: 0 → 32

Now the scheduler tells the worker: prefill the new tokens
(T3_user, the part not in cache) at physical positions 32..47.

Worker writes K at:

  physical + position_offset = 32 + 32 = 64  →  K at logical 64
  physical + position_offset = 47 + 32 = 79  →  K at logical 79

WAIT. We're writing new K at logical 64..79. But slot 16..31
ALREADY has K at logical 80..95 — wait, no.

Let me re-look. Survivors at slot 16..31 are at logical 80..95.
New writes at slot 32..47 are at logical 64..79.

  Slot   0..15: K at logical  0..15
  Slot  16..31: K at logical 80..95   ← SURVIVORS
  Slot  32..47: K at logical 64..79   ← NEW WRITES — out of order!

The K cache logical positions are now NON-MONOTONIC across
slots. Slot 16 (logical 80) is HIGHER than slot 32 (logical 64).

For attention, this is weird but not yet broken. The actual
collision happens during decode.

Decode T3_assistant. The decode step writes K at:

  physical = 48, offset = 32, logical = 80

But slot 16 ALREADY has K at logical 80! Two different K
vectors at the SAME logical position 80.

When Q at some later position attends to the cache, it computes:
  RoPE_relative(Q_logical, K_logical) for each K

For Q at logical 100 (during decode) attending K[slot=16, logical=80]
and K[slot=48, logical=80]: both K are at the SAME relative position 20
from Q. RoPE rotates them by the same amount. The attention sums
both into the same "position 20 ago" — but they're DIFFERENT TOKENS.

The model sees DUPLICATE POSITION 80 in its context. Its attention
math is corrupted. Output is garbage-but-confident (softmax always
produces some distribution; the model picks the most likely garbage).

For our smoke run, this manifested as:
  vLLM emits "The capital of Italy?" when user asked about Germany,
  with logprob = -0.06 (94% confidence in this wrong answer).


-----------------------------------------------------------
WHY DOES THIS HAPPEN? The single position_offset can't express it.
-----------------------------------------------------------

To AVOID the collision, the new writes need to start at logical
positions HIGHER than any inherited K. Specifically: highest
survivor logical = 95. So new writes need to start at logical >= 96.

The required position_offset would be:
  offset = max_survivor_logical + 1 - num_computed_post
        = 96 - 32
        = 64

NOT 32 (which is what vLLM computed by just adding tokens_evicted=32
to the starting offset of 0).

For this offset=64 to work correctly:
  Slot   0..15: K at logical  0..15 (sys). Q must use offset=0
                                         (so Q logical matches K logical).
  Slot  16..31: K at logical 80..95. Q must use offset=64
                                     (16+64=80, 31+64=95 ✓).
  Slot  32..47: K at logical 96..111 (new writes). Q uses offset=64
                                     (32+64=96, 47+64=111 ✓).
  Slot  48..63: K at logical 112..127 (decode). Q uses offset=64.

Two different offsets needed within one request:
  - offset 0 for sys
  - offset 64 for everything else (the survivors AND new writes)

vLLM currently stores ONE offset per request and applies it to
ALL of the request's tokens. That can't express "0 for some,
64 for others". Hence the bug.


-----------------------------------------------------------
THE FIX OPTIONS
-----------------------------------------------------------

Option A: "drop survivors" (current shipped fix)
  At admission, free the survivor blocks (slot 16..31 in turn 4's
  example). Re-prefill them. After fix:
    Slot   0..15: K at logical  0..15  (sys)
    Slot  16..47: K at logical 16..47 (new prefill of T1+T2 tail
                                       + T3_user, at offset 0)

  Wait — but then the offset would be 0 again? Let me re-check.
  Actually with offset=32 bumped, slot 16..47 prefilled writes K
  at logical 48..79. Cache:
    Slot   0..15: logical  0..15
    Slot  16..47: logical 48..79 (re-prefilled at offset 32)
  Monotonic. No collision. Loses prefix-cache reuse for the
  inherited T2_user block (slot 48..63 in old layout).

  ✓ Correct
  ✗ Loses prefix-cache reuse on chained admission

Option B: 2-piece position_offset (the better fix)
  Track protected_prefix_len per request (= sys boundary).
  Bump offset SMARTLY: position_offset = max(old + total_evicted,
                                             max_survivor_logical + 1
                                             - num_computed_post).
  GPU compute: Q logical = physical + offset if physical >=
                            protected_prefix_len else physical.

  After fix:
    Slot   0..15: K at logical  0..15 (sys), offset=0
    Slot  16..31: K at logical 80..95 (T2 user, inherited), offset=64
    Slot  32..47: K at logical 96..111 (new prefill), offset=64
    Slot  48..63: K at logical 112..127 (decode), offset=64
  Monotonic. No collision. PRESERVES the inherited K at slot 16..31.

  ✓ Correct
  ✓ Keeps prefix-cache reuse


-----------------------------------------------------------
TL;DR
-----------------------------------------------------------

The bug: vLLM's prefix cache lets a new request inherit K vectors
written by a prior request that had a non-zero position_offset.
Those K's logical positions are FROZEN. When the new request also
runs admission (chained admission), its naive position_offset bump
puts new writes at LOGICAL positions that COLLIDE with inherited
survivor K positions. Two K at the same logical position → garbage
attention.

The root cause: vLLM uses ONE position_offset per request. After
chained admission, the request's cache needs TWO offsets — one
for sys (frozen at write-time offset 0) and one for everything
else (frozen at the donor's offset, which is > 0).

The simplest fix that preserves prefix-cache reuse: track sys
boundary per request (protected_prefix_len), compute a smart
new offset on admission, and apply offset only to physical
positions >= protected_prefix_len.

For TURN-WISE admission (the case we care about), the
"everything else" always shares ONE offset value, so just two
pieces (sys vs post-sys) suffice. We don't need general
N-piece piecewise.
